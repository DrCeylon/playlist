# Format JSON des playlists

*Le contrat de données — mon côté PolicyCenter qui ressort.*

## Philosophie

Comme une **police d'assurance**, une playlist a :
- un **numéro** (`name`) ;
- une **description** (`description`) ;
- des **garanties** (les sections) ;
- des **éléments couverts** (les morceaux).

Si le JSON est invalide, le loader refuse — message d'erreur en français, pas de `KeyError` cryptique.

## Schéma complet

```json
{
  "name": "🏝 Orlando Pool Party 2026",
  "description": "Pool party à Orlando : montée progressive, bonne humeur maximale, sans reggaeton.",
  "sections": [
    {
      "name": "🌴 Warm Up Tropical",
      "songs": [
        {"artist": "Kygo", "title": "Firestone"},
        {"artist": "Lost Frequencies", "title": "Reality"}
      ]
    },
    {
      "name": "☀️ Sunny Vibes",
      "songs": [
        {"artist": "Harry Styles", "title": "As It Was"}
      ]
    }
  ]
}
```

## Champs obligatoires

| Champ | Type | Règle |
|-------|------|-------|
| `name` | `string` | Non vide. Nom de la playlist dans Apple Music |
| `sections` | `array` | Liste de sections. Peut être vide (playlist vide) |

## Champs optionnels

| Champ | Type | Défaut |
|-------|------|--------|
| `description` | `string` | `"Generated by Apple Music Playlist Builder"` |

## Structure d'une section

| Champ | Type | Règle |
|-------|------|-------|
| `name` | `string` | Nom affiché. Défaut : `"Playlist"` |
| `songs` | `array` | Liste ordonnée de morceaux |

## Structure d'un morceau

| Champ | Type | Règle |
|-------|------|-------|
| `artist` | `string` | Non vide après trim |
| `title` | `string` | Non vide après trim |

## Règles de validation

Le loader (`playlist_builder/playlists/loader.py`) vérifie :

- JSON syntaxiquement valide
- Racine = objet (pas un tableau)
- `name` présent et non vide
- `sections` = liste
- Chaque section = objet avec `songs` = liste
- Chaque morceau = objet avec `artist` et `title` non vides

### Exemples d'erreurs

```
Playlist invalide: Champ manquant 'title' pour le morceau #3 dans la section '🌴 Warm Up Tropical'.
```

```
Playlist invalide: Le champ 'name' est requis et ne peut pas être vide.
```

## Ordre des sections

**L'ordre est contractuel.**

1. Les sections sont traitées dans l'ordre du tableau `sections`
2. Les morceaux sont traités dans l'ordre du tableau `songs` de chaque section
3. La playlist Apple Music finale suit cet ordre

C'est aussi le contrat stable pour la future **app iOS**.

## Clé unique d'un morceau

En interne, chaque morceau a une clé :

```
{artiste_normalisé}::{titre_normalisé}
```

Exemple : `kygo::firestone`

Utilisée pour la détection de doublons.

## Bonnes pratiques

### Nommage des sections

Utilise des noms qui racontent une histoire :

```
🌴 Warm Up Tropical    →  début doux
☀️ Sunny Vibes         →  montée
🎉 Everybody In The Pool →  peak
🌙 Night Pool Finale   →  finish fort
```

Comme les **étapes d'un parcours sinistre** — chacune a un rôle, un ordre, un sens.

### Artistes avec featuring

Préfère l'artiste principal tel qu'il apparaît sur Apple Music :

```json
{"artist": "Mark Ronson", "title": "Uptown Funk"}
```

Le scoring gère les variantes `feat.` partiellement.

### Caractères spéciaux

Les titres avec apostrophes, guillemets ou accents sont supportés. Le JSON UTF-8 est obligatoire.

### Exclusions explicites

Documente tes choix dans `description` :

```json
"description": "Sans reggaeton. Sans ballades tristes. Papa a décidé."
```

## Modèle Python correspondant

```python
@dataclass
class TrackRef:
    artist: str
    title: str
    section: str

@dataclass
class PlaylistDefinition:
    name: str
    tracks: list[TrackRef]      # aplatissement ordonné des sections
    description: str
```

Chaque `TrackRef` conserve son `section` d'origine pour l'affichage groupé.
